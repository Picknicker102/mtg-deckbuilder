[Titel]
MTG Commander Hinweis-Wrapper v8.9-minJSON (stabil)

[Zweck]
Praxis-Begleiter für den Deckbuilder. Erklärt Anwendung von CORE + mtg_master.json im Alltag. Ändert keine Regeln. Wenn CORE und Wrapper kollidieren, gewinnt immer der CORE.

[Projektdateien]
- MtgBuilder_Core.txt (Regeln, Hardguards, Snapshot-first)
- backend/data/mtg_master.json (oracle_overrides, alias_map, banned_snapshot; keine Sammlung, keine Box-/Fachlabels)
- MTG Hinweise wrapper.txt (dieser Begleiter)
- Mtg Builder User Guide.txt (UI/Coaching, keine Regelmacht)
Details zu Struktur und Feldern siehe CORE-Abschnitt "Dateien / Struktur".

[Priorität und Rollen]
- CORE > Wrapper > User Guide > internes Wissen/Web
- Snapshot-first: banned_snapshot.cards & oracle_overrides schlagen Web/RC immer (Details: CORE "Alias-, Override- & Snapshot-Regeln")
- alias_map wird vor Web-Resolve angewendet

[JSON in der Praxis]
- JSON muss alle drei Kernobjekte haben (oracle_overrides, alias_map, banned_snapshot). Fehlt eins -> Fail-Closed, kurze Fehlermeldung.
- JSON ist kein Kartenspeicher. Nur Aliasse, Overrides, Bann-Snapshot. Keine persönlichen Listen, keine Lager-/Boxlabels.
- Pfadreferenz: backend/data/mtg_master.json

[Bannlogik kompakt]
- Wenn banned_snapshot.cards[name] = true -> Karte sofort verwerfen.
- Nicht gelistet = vorläufig legal; optionale Online-Checks je rc_mode.
- Web/RC dürfen den Snapshot nie überstimmen. Sie dienen nur als Ergänzung bei Lücken/Fehlern (siehe CORE "Bann- & Legalitätslogik").

[Namensauflösung kompakt]
- alias_map anwenden (z. B. "Sonic" -> "Sonic the Hedgehog").
- oracle_overrides priorisieren bei CI/is_commander/commander_legal/Tags.
- Bei Override- und Web-Konflikten gilt: Override gewinnt. Details im CORE-Abschnitt "Alias-, Override- & Snapshot-Regeln".

[Parameter / Modi Praxis]
- Defaults: browsing=on, rc_mode=hybrid, power_bracket=4, allow_loops=auto.
- rc_mode nur für Legalitätschecks; browsing steuert Web generell. (Referenz: CORE "Steuer-Parameter & Modi")
- Bei Tool-/Web-Limits: weiterbauen mit Snapshot-first, klar benennen was offline geprüft wurde.

[Ausgabe & Hardguard]
- Immer finaler 100/100-Check auf dem Antworttext: 100 Kartenzeilen + 1 Validierungszeile.
- Validierungszeile exakt:
  Validation: 100/100✔️ RC-Snapshot✔️ RC-Sync AB (Modus: <strict|hybrid|offline>)✔️ Commander-legal✔️ CI✔️ Moxfield-ready✔️
- Keine Ausgaben bei Fehl-Checks; kurze Meldung statt halber Decks. Details: CORE "Interne Validierung" und "Ausgabeformate".

[Praxis-Selbstchecks vor Ausgabe]
- Zeilenzählung, Duplikate (non-Basics nur 1x), Banncheck (Snapshot), CI-Konsistenz, Commander-Zeile prüfen.
- Checks immer auf finalem Rohtext, nicht nur internen Tools. Bei Fehler -> kein Deck, klare Fehlermeldung. Siehe CORE Validierungsabschnitt.

[Zusammenspiel mit User Guide]
- User Guide darf Analyse/Statistiken liefern, aber nie die 100/100-Form oder Validierungszeile ändern.
- Wrapper beschreibt nur das erwartete Verhalten; Regelhoheit bleibt beim CORE.

[Hinweis Sammlung]
- Diese Datei und mtg_master.json enthalten keine persönliche Kartensammlung und keine Box-/Fachlabels.
